# Skill Auditor

审计目标：在安装或使用 skill 之前，判断它到底是：
- **本体安全、可安装**
- **无明显恶意，但权限/影响过大，需要收紧后再用**
- **存在明显恶意或高危信号，不建议安装**

优先做保守判断。不要因为 skill 名字看起来官方、流行、下载多，或自称安全，就降低审慎标准。

## 核心原则

1. **先盘点，再结论**
   先列出 skill 全部文件，再判断风险。不要只看 `SKILL.md` 标题或简介。

2. **区分“恶意”与“高影响”**
   这两类风险必须分开写：
   - **恶意/脏代码风险**：例如窃取凭据、隐蔽外联、提权、下载执行、持久化、规避检测、自修改。
   - **高权限/高影响风险**：例如大量移动文件、递归删除、批量改写仓库、复用浏览器登录态、接管认证令牌。
   前者通常直接判为 **高** 或 **极高**；后者可能是 **中** 或 **高**，并给出限制使用条件。

3. **报告必须解释原因**
   不允许只给“低/中/高”结论，必须说明：
   - 触发了哪些具体信号
   - 为什么这些信号会提高风险
   - 它属于恶意风险、权限风险、影响风险，还是多者叠加

4. **报告必须给可执行建议**
   每份报告都要给出清晰动作，而不是空泛地说“谨慎使用”。例如：
   - 改为默认 dry-run
   - 对关键目录增加二次确认
   - 限制只允许在指定路径运行
   - 移除远程下载执行
   - 拆掉不必要权限
   - 明确 provenance
   - 补充回滚机制
   - 改为 first-class tools 优先

5. **工具优先，不被 shell 示例牵着走**
   如果 skill 提供 shell 命令示例，不要默认照抄执行。优先用现有一等工具（如 `read`、`write`、`edit`、平台原生工具）完成审计。

6. **区分“公开页校准”与“源码级终审”**
   - 仅基于 ClawHub、GitHub 页面、第三方介绍页做判断时，要明确标注：这是**公开说明级评估**，不是完整源码终审。
   - 如果风险较高、边界敏感、或用户准备真实安装，优先建议把 skill 文件**完整拉到本地**，逐文件做 thorough 审计。
   - 审完后要给出处理建议：**保留样本继续研究**，还是**删除本地副本以减少暴露面**。

7. **供应链独立判断**
   “官方”“热门”“带 star”“被转发很多”都不能代替代码审计。来源链、作者、镜像关系、提交指向是否自洽，必须单独判断。

## 审计流程

1. **建立清单**
   - 列出 skill 全部文件
   - 标出 `scripts/`、`references/`、`assets/`、隐藏文件、二进制、配置文件
   - 说明哪些文件已实际检查

2. **做来源审查**
   - 来源是本地内置、官方仓库、fork、镜像，还是陌生仓库
   - `_meta.json`、README、提交链接、作者名、仓库路径是否相互一致
   - 是否存在“页面写 official，但来源链不干净”的情况

3. **做规范审查**
   - frontmatter 是否只保留必要字段
   - description 是否清楚描述触发条件
   - 是否把触发逻辑错误地塞进正文
   - 是否乱塞 README、CHANGELOG、安装说明等冗余文件
   - 是否过度营销、社工式描述、用词夸张

4. **做行为审查**
   - 是否会执行脚本、下载代码、安装依赖、调用远程 API、修改系统配置
   - 是否读取敏感文件、浏览器 cookie、token、SSH key、环境变量、账号凭据
   - 是否批量修改本地文件、仓库、数据库、文档结构
   - 是否涉及浏览器自动化、认证集成、会话复用、绕过检测

5. **做风险归因**
   审计结论必须明确归因到以下一种或多种：
   - 恶意/脏代码风险
   - 供应链/来源风险
   - 权限过宽风险
   - 高影响本地操作风险
   - 说明不透明/规范不足风险
   - 滥用增强/规避检测风险

6. **输出正式报告**
   默认跟随用户提问语言输出；若用户是中文，就用中文标题和字段名。技术术语可保留英文，但要配简短中文说明。

7. **对高风险但非恶意的 skill，单独给“改造建议 + 是否请示”**
   如果 skill 主要问题不是恶意，而是权限过宽、影响过大、默认行为太激进，那么报告中要额外说明：
   - 是否值得改造后再用
   - 可以怎么改造（缩小范围、改默认行为、加确认、加 dry-run、加回滚、拆权限）
   - 是否应在真正改造前先向用户请示

## 风险等级定义

统一使用中文五档：
- **极低**：几乎纯文本、来源清楚、规范好、无执行与高影响操作
- **低**：整体较安全，可能有轻微规范或来源瑕疵，但无明显恶意或高影响行为
- **中**：无明显恶意，但权限、影响范围、敏感领域或来源问题值得注意；可用，但应先加限制
- **高**：存在明显高危信号，或恶意迹象与高影响操作叠加；不建议直接安装
- **极高**：存在强烈恶意/滥用/隐蔽执行/提权/窃密/规避检测组合信号；不建议安装

### 判定优先级

如果发现以下任一情况，优先把等级抬到 **高** 或 **极高**：
- 隐蔽外联或疑似窃取数据
- 下载后执行、远程拉取脚本并运行
- 提权、持久化、自更新、自修改
- 读取或导出凭据、cookie、token、SSH key
- 明确强调 stealth / evasion / undetectable / captcha bypass / proxy rotation

如果没有明显恶意，但存在以下情况，通常判为 **中** 到 **高**：
- 大范围本地文件改动
- 递归移动/删除/改名
- 浏览器登录态复用
- 批量改写仓库或文档结构
- 认证、令牌、密钥等敏感领域操作

## 输出要求

报告至少包含以下字段，并按下面顺序组织，确保使用者先看到结论、再看到行动项、最后再看解释细节：
- **技能审计报告**
- 技能
- 来源
- 作者
- 版本
- 已检查文件
- **风险等级**
- **风险类型**
- **是否建议安装**
- **是否建议拉到本地做彻底审计**
- **审计后文件处理建议（保留 / 删除）**
- **安全建议**
- **如需改造：改造建议与是否请示**
- 判定原因
- 清单
- 规范检查
- 行为检查
- 风险信号

### 推荐展示顺序

1. **风险等级**：先让用户一眼看到结论
2. **风险类型**：立刻说明这是恶意风险、权限风险、影响风险还是来源风险
3. **是否建议安装**：给出安装层面的明确动作
4. **是否建议拉到本地做彻底审计**：告诉用户要不要升级到源码级终审
5. **审计后文件处理建议**：说明是保留样本、隔离保留，还是直接删除本地副本
6. **安全建议**：优先给出能立刻执行的改法或限制条件
7. **如需改造：改造建议与是否请示**：针对高风险但非恶意的 skill，明确是否值得改造、是否应先请示
8. **判定原因**：解释为什么是这个等级
9. **详细检查内容**：把清单、规范检查、行为检查、风险信号放在后面作为证据

### “是否建议安装”口径

- **极低 / 低**：可安装
- **中**：可安装，但建议先限制权限、范围或默认行为
- **高**：谨慎，不建议直接安装
- **极高**：不建议安装

## Head-to-head 比较

如果用户让你比较两个 skill：
- 分别输出两份简版风险结论
- 明确指出 **更安全的选择**
- 说明 **为什么更安全**
- 说明 **较弱的一方有哪些值得借鉴的点**
- 最后给出 **综合建议**

## 做 replacement skill 的规则

如果用户想基于两个现有 skill 做一个更安全的新 skill：
- 先指出原有两者各自优点和风险
- 新 skill 应比旧 skill **更窄、更透明、更少权限**
- 能不用脚本就不用脚本
- 必须先补审计缺口，再增加功能
- 不要为了“更强”而扩大攻击面

## 重要提醒

- 不要把“文案规范”误判成“绝对安全”
- 也不要把“能力很强”直接等同于“恶意”
- 关键是分清：**是不是坏**，和 **是不是权力太大**
- 结论要让用户一眼看懂：为什么危险，危险在哪，能不能通过配置降低风险

# 审计清单与报告模板

## 一、文件清点

先列出并检查：
- `SKILL.md`
- `_meta.json`
- `scripts/`
- `references/`
- `assets/`
- 隐藏文件
- 可执行文件 / 二进制
- 安装脚本 / 构建脚本 / 下载脚本

检查时回答：
- 一共多少文件？
- 实际读了哪些文件？
- 有没有没读但可能高风险的文件？

## 二、来源与供应链检查

回答这些问题：
- 来源是本地内置、官方仓库、fork、镜像还是陌生仓库？
- 作者、仓库名、页面文案、`_meta.json` 是否一致？
- commit 链接是否指向另一个 repo / owner？
- 是否存在“看起来官方，但 provenance 混杂”的情况？
- 是否只靠 stars/downloads 建立信任？如果是，降级信任。

## 三、规范检查

检查：
- frontmatter 是否只含必要字段
- description 是否承担了触发说明
- 正文是否错误地承载 “When to Use” 触发逻辑
- 是否夹带 README / CHANGELOG / INSTALLATION_GUIDE 之类冗余文件
- 是否存在营销式、社工式、夸张承诺
- 是否过宽触发，导致容易误用

## 四、行为检查

检查 skill 是否会：
- 执行脚本
- 下载并执行远程内容
- 安装依赖
- 发起外联
- 读取环境变量或敏感文件
- 操作浏览器 profile / cookies / sessions
- 处理 token / API key / SSH key / 凭据
- 批量移动、删除、改名本地文件
- 修改系统配置、启动项、计划任务、守护进程
- 绕过检测、规避风控、处理验证码、轮换代理

## 五、影响面检查

重点判断：
- 是只读还是会写入？
- 写入范围是局部还是大范围？
- 是否支持 dry-run？
- 是否支持 undo / rollback？
- 是否默认先执行，而不是先预览？
- 是否对高风险目录有二次确认？
- 失败时是否可能留下半完成状态？

## 六、风险类型归因

至少从这些类型中归类：
- 恶意/脏代码风险
- 供应链/来源风险
- 权限过宽风险
- 高影响本地操作风险
- 说明不透明/规范不足风险
- 滥用增强/规避检测风险

## 七、风险等级判定建议

### 极低
- 纯文本或近乎纯文本
- 来源清楚
- 无脚本、无外联、无高影响操作
- 规范完整

### 低
- 无明显恶意
- 可能存在轻微来源或规范问题
- 但没有高影响默认行为

### 中
- 无明显恶意
- 但涉及敏感领域、较高权限、较大影响范围，或来源链略混乱
- 经过限制配置后可用

### 高
- 存在明显高危信号
- 或恶意迹象、高权限、高影响叠加
- 不建议直接安装

### 极高
- 强烈恶意或滥用导向
- 包含窃密、提权、隐蔽执行、规避检测组合特征
- 不建议安装

## 八、报告模板（中文）

```markdown
# 技能审计报告
- 技能：<name>
- 来源：<source>
- 作者：<author>
- 版本：<version>
- 已检查文件：<files>

## 风险等级
- <极低 / 低 / 中 / 高 / 极高>

## 风险类型
- <恶意/脏代码风险 | 供应链/来源风险 | 权限过宽风险 | 高影响本地操作风险 | 说明不透明/规范不足风险 | 滥用增强/规避检测风险>

## 是否建议安装
- <可安装 / 可安装，但建议先限制权限或调整配置 / 谨慎，不建议直接安装 / 不建议安装>

## 是否建议拉到本地做彻底审计
- <建议 / 可选 / 不需要>
- 说明为什么需要或不需要做源码级终审

## 审计后文件处理建议
- <保留样本继续研究 / 隔离保留 / 审后删除本地副本>
- 说明理由

## 安全建议
- 给出明确可执行动作
- 不少于 3 条，且尽量能直接实施

## 如需改造：改造建议与是否请示
- 是否值得改造后再用
- 可以改造哪些默认行为、权限边界或回滚机制
- 是否应先向用户请示再改造

## 判定原因
- 具体说明为什么是这个等级
- 明确是“代码本身恶意”还是“本身不恶意但权限/影响过大”
- 如果两者都有，要明确写叠加关系
- 如果当前只基于公开页面/说明判断，要明确写“这不是完整源码终审”

## 清单
- ...

## 规范检查
- ...

## 行为检查
- ...

## 风险信号
- ...
```

### 模板顺序说明

把 **风险等级** 和 **风险类型** 放在最上面，是为了让使用者先快速建立判断；
把 **是否拉到本地做彻底审计** 和 **审后文件处理建议** 提前，是为了帮助使用者决定下一步操作；
把 **安全建议** 和 **改造建议** 提前，是为了让使用者先知道该怎么做；
再把 **判定原因** 和详细检查内容放后面，作为支撑证据。

## 九、比较两个 skill 的附加模板

```markdown
## 对比结论
- 更安全的选择：<skill>
- 原因：...
- 可借鉴点：...
- 综合建议：...
```

## 十、语言规则

- 默认跟随用户提问语言输出
- 中文用户优先输出完整中文标题与字段名
- 技术术语可保留英文，但要配简短中文说明
- 用户明确要求双语时，再输出简洁双语版
